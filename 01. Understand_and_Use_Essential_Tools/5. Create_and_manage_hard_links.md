## 🎯 Objective

Understand how hard links work internally in Linux, how they relate to inodes, and how they are used in troubleshooting, recovery, and filesystem management.

---

- Every file in Linux is associated with an inode
- An inode stores metadata such as permissions, ownership, timestamps, size, and disk block locations
- Filenames are not stored inside the inode
- A hard link is an additional filename pointing to the same inode
- Hard links share the same inode number and data blocks
- Creating a hard link does not create a copy of the file
- Changes made through one hard link are visible through all linked names
- Deleting one hard link does not remove the underlying data if other hard links exist
- Linux deletes file data only when the link count reaches zero and no process is using the file
- Hard links cannot cross filesystem boundaries
- Hard links cannot normally be created for directories
- Link count represents the number of directory entries referencing an inode

---

# 💻 Commands Used

```bash
ln file1 file2      # Creates a hard link
ls -li             # Displays inode numbers and link count
stat file1         # Shows detailed inode metadata
rm file1           # Removes one hard link entry
cat file1          # Reads file content
```

---

# Troubleshooting Notes

### 1. Original file deleted but data still exists

Cause:

Another hard link still points to the same inode.

Verification:

```bash
ls -li
```

Fix:

Access the file through the remaining hard link.

---

### 2. Disk space not released after file deletion

Cause:

A running process still holds the inode open.

Verification:

```bash
lsof | grep deleted
```

Fix:

Restart or stop the process holding the file.

---

### 3. Invalid cross-device link

Cause:

Attempting to create a hard link across filesystems.

Example:

```bash
ln /home/file1 /mnt/file2
```

Fix:

Use a symbolic link instead.

---

### 4. File content changes unexpectedly

Cause:

Multiple hard links reference the same inode.

Verification:

```bash
ls -li
```

Check for identical inode numbers.

# Real-World Production Scenarios

### 1. Configuration Recovery

A critical configuration file is accidentally deleted.

A hard link backup exists.

Result:

Application data remains available through the remaining hard link.

---

### 2. Log File Investigation

A large log file is deleted but disk space remains consumed.

Cause:

The application process still holds the inode open.

Investigation:

```bash
lsof | grep deleted
```

---

### 3. Application Still Reading Deleted File

A file is removed from the filesystem but the application continues working.

Reason:

The process has an active file descriptor referencing the inode.

---

### 4. Storage and Backup Systems

Backup mechanisms may use hard links to avoid storing duplicate copies of identical data.

Benefit:

Reduced storage consumption.

---

### 5. Linux Filesystem Troubleshooting

Administrators use inode numbers and link counts to investigate unexpected file behavior and storage issues.

Useful Commands:

```bash
ls -li
stat filename
```

---

# Key Takeaways

- Hard Link = Same Inode
- Multiple filenames can reference the same data
- Hard links are not copies
- Deleting one hard link does not delete the file
- Data is removed only when link count becomes zero
- Hard links cannot cross filesystems
- Understanding inodes is essential for Linux administration and troubleshooting
